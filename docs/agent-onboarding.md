# AI Agent 작업 안내

이 문서는 AI Agent로 AI Commit Advisor 프로젝트를 작업할 때 참고하는 안내입니다. 매번 같은 규칙을 프롬프트에 붙이지 않아도 되도록, 이 프로젝트의 문서화와 검증 기준을 한곳에 모았습니다.

## 이 문서를 먼저 읽어야 하는 이유

AI Agent 코딩은 단순히 "코드를 대신 작성해주는 도구"를 쓰는 일이 아닙니다. 프로젝트의 구조, 검증 방식, 문서화 기준, 팀의 작업 순서를 Agent가 이해해야 결과물을 리뷰하고 유지보수하기 쉬워집니다.

이 프로젝트는 `AGENTS.md`, `ROADMAP.md`, `AI_CHANGELOG.md` 같은 문서로 Agent의 작업 방식을 관리합니다. 처음에는 문서가 많아 보일 수 있지만, 역할은 단순합니다.

- `AGENTS.md`: Agent가 이 저장소에서 따라야 하는 작업 규칙입니다.
- `ROADMAP.md`: 의미 있는 기능, UX, 문서, 운영 작업의 진행 상태를 추적합니다.
- `AI_CHANGELOG.md`: Agent가 실제로 바꾼 내용과 검증 결과를 기록합니다.

## AGENTS.md란 무엇인가요?

`AGENTS.md`는 Codex 같은 코딩 Agent에게 알려주는 프로젝트 작업 규칙입니다. 사람이 매번 프롬프트에 길게 설명하지 않아도, Agent가 이 저장소에서 일할 때 지켜야 할 기준을 담아 둡니다.

이 프로젝트의 `AGENTS.md`에는 다음과 같은 규칙이 들어 있습니다.

- 의미 있는 변경 후 `AI_CHANGELOG.md`를 업데이트한다.
- 기능, UX, 문서, 운영 작업을 시작하기 전에 `ROADMAP.md`를 확인한다.
- DB schema 변경은 Alembic migration으로 처리한다.
- 사용자-facing 문서는 한국어 설명을 우선한다.
- 변경 성격에 따라 어떤 문서를 함께 확인하거나 갱신해야 하는지 판단한다.
- 검증은 변경 내용에 맞는 가장 저렴한 표면에서 수행한다.

즉, `AGENTS.md`는 "이번 작업의 요구사항"이 아니라 "이 프로젝트에서 일하는 방식"에 가깝습니다.

## AGENTS.md를 매번 프롬프트에 써야 하나요?

아니요. Codex를 이 프로젝트 폴더에서 실행하면 `AGENTS.md`는 작업 시작 시 자동으로 읽힙니다. 따라서 `ROADMAP.md` 확인, `AI_CHANGELOG.md` 업데이트 같은 반복 규칙을 매번 프롬프트에 쓸 필요는 없습니다.

프롬프트에는 이번 작업의 목표를 구체적으로 적으면 됩니다.

좋은 예:

```text
프로젝트 선택 화면에서 최근 사용한 저장소를 먼저 보여주게 해줘.
```

```text
분석 결과 저장 로직에 대한 테스트를 추가해줘.
```

```text
README의 Quick Start가 현재 실행 방식과 맞는지 점검하고 고쳐줘.
```

필요할 때만 덧붙이면 좋은 문장:

```text
프로젝트 규칙에 맞게 필요한 문서 업데이트까지 확인해줘.
```

```text
ROADMAP에 이미 있는 작업이면 상태도 같이 갱신해줘.
```

## ROADMAP.md 확인도 매번 말해야 하나요?

보통은 매번 말하지 않아도 됩니다. 이 프로젝트의 `AGENTS.md`가 의미 있는 기능, UX, schema, test, 문서 작업 전에 `ROADMAP.md`를 확인하도록 지시하고 있기 때문입니다.

다만 다음 상황에서는 프롬프트에 명시하면 더 안전합니다.

- 로드맵에 이미 있는 작업을 이어서 하는 경우
- 여러 작업 후보 중 어떤 항목을 먼저 할지 정해야 하는 경우
- 작업 완료 후 로드맵 상태를 반드시 `Done`으로 바꾸고 싶은 경우
- commit hash나 changelog 연결까지 정리해야 하는 경우

예:

```text
ROADMAP의 "AI Agent Onboarding Guide" 작업을 이어서 완료해줘.
구현, 문서 링크, AI_CHANGELOG 업데이트, 검증까지 끝나면 상태를 Done으로 바꿔줘.
```

## Codex가 자동으로 읽는 것과 직접 말해야 하는 것

Codex가 프로젝트 폴더에서 실행될 때 자동으로 읽는 것은 주로 프로젝트 작업 규칙입니다.

- `AGENTS.md`의 durable instruction
- 현재 작업 폴더 기준으로 적용되는 상위 또는 하위 instruction 파일
- 이미 대화 중에 제공된 목표와 맥락

반대로 다음 내용은 프롬프트에 직접 적는 편이 좋습니다.

- 이번에 원하는 구체적인 결과
- 사용자가 이미 알고 있는 버그 재현 절차
- 반드시 유지해야 하는 화면 문구나 업무 표현
- 이번 작업에서 제외할 범위
- 선호하는 검증 방법이나 실행 환경
- 리뷰에서 특히 확인하고 싶은 위험 요소

좋은 프롬프트는 길 필요가 없습니다. 다만 "무엇을 바꾸고 싶은지", "어떤 상태가 완료인지", "주의할 조건이 있는지"가 분명해야 합니다.

## 다른 AI 도구를 쓸 때는 어떻게 하나요?

Codex가 아닌 도구는 `AGENTS.md`를 자동으로 읽지 않을 수 있습니다. 이 경우에는 다음 중 하나를 선택하세요.

- `AGENTS.md` 파일을 첨부한다.
- 필요한 규칙만 프롬프트에 요약해서 넣는다.
- 작업 전에 "이 저장소의 `AGENTS.md`를 먼저 읽고 따라줘"라고 요청한다.

도구마다 파일 접근 방식과 instruction 처리 방식이 다르므로, `AGENTS.md`를 자동으로 읽는지 확실하지 않다면 직접 확인하는 것이 안전합니다.

## 자주 쓰는 요청 예시

기능 수정:

```text
Program Detail 화면에서 저장된 구현상태 분석 결과가 없을 때 안내 문구를 더 자연스럽게 바꿔줘.
테스트가 필요한 변경이면 focused test도 추가해줘.
```

문서 점검:

```text
README와 Setup 문서의 Docker 실행 설명이 현재 docker-compose.yml과 맞는지 확인하고 필요한 부분만 고쳐줘.
```

버그 수정:

```text
Project Chat에서 현재 HEAD와 Indexed HEAD가 같은데도 재인덱싱 경고가 보이는 문제를 확인하고 고쳐줘.
재현 가능한 테스트가 있으면 추가해줘.
```

리뷰:

```text
현재 변경사항을 코드리뷰 관점으로 봐줘.
버그, 회귀 위험, 빠진 테스트를 먼저 말해줘.
```

작업 범위 제한:

```text
이번에는 코드 변경 없이 원인만 조사해줘.
파일을 수정하지 말고 어떤 파일을 봤는지만 정리해줘.
```

## 사람이 마지막에 확인해야 할 것

AI Agent가 코드를 바꾸더라도 최종 책임은 사람이 가집니다. 특히 다음은 직접 확인하는 것이 좋습니다.

- 변경 diff가 요청 범위를 벗어나지 않았는지
- 테스트나 검증 명령이 실제로 실행됐는지
- `AI_CHANGELOG.md`에 변경 내용과 검증 결과가 남았는지
- 사용자-facing 문서의 한국어 표현이 자연스러운지
- LLM, embedding, Docker, DB처럼 환경 영향을 받는 기능이 올바른 표면에서 검증됐는지
- Agent가 추측한 업무 요구사항이 실제 팀 기준과 맞는지

## 추천 사용 흐름

1. 작업 전, 원하는 결과를 짧고 구체적으로 작성합니다.
2. Codex가 제안하거나 수행한 변경 범위를 확인합니다.
3. 테스트와 문서 업데이트가 필요한 작업인지 확인합니다.
4. Agent가 제시한 검증 결과를 보고, 중요한 화면이나 업무 흐름은 사람이 한 번 더 확인합니다.
5. commit 전에는 diff, `AI_CHANGELOG.md`, 필요 문서, 로드맵 상태를 확인합니다.

이 흐름을 따르면 개발자 각자가 Agent를 쓰더라도 프로젝트의 기록, 검증, 문서화 방식이 크게 흔들리지 않습니다.
